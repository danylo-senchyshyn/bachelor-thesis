\abstract{%
  % english
  This thesis presents a comparative study of two architecturally distinct implementations of the same reservation system — a monolithic and a microservice-based version — developed using Java, Spring Boot, and PostgreSQL. The monolithic prototype implements all domain responsibilities within a single deployable unit with a unified transactional boundary, while the microservice prototype decomposes the system into autonomous services coordinated through event-driven choreography via Apache Kafka and the transactional outbox pattern.
  Performance measurements using the k6 load testing tool showed that the monolithic architecture achieves 14\,\% higher peak throughput (1\,110 vs.\ 972 req/s), 2.2× lower median HTTP latency, and a dramatically shorter end-to-end processing time (5\,ms vs.\ 5\,210\,ms median). The dominant factor behind the end-to-end difference is the asynchronous outbox scheduler, not raw compute performance. Qualitative analysis confirmed that microservices provide better fault isolation, independent deployability, and potential for selective scaling at the cost of significantly higher operational complexity. The findings support context-dependent architectural decision-making rather than a universal preference for either approach.
}{%
  % slovak
  Táto práca predstavuje porovnávaciu štúdiu dvoch architektonicky odlišných implementácií rovnakého rezervačného systému — monolitickej a mikroservisnej verzie — vyvinutých v prostredí Java, Spring Boot a PostgreSQL. Monolitický prototyp implementuje všetky doménové zodpovednosti v rámci jedného nasaditeľného artefaktu s jednotnou transakčnou hranicou, zatiaľ čo mikroservisný prototyp dekomponuje systém na autonómne služby koordinované prostredníctvom event-driven choreografie cez Apache Kafka a vzor transactional outbox.
  Výkonnostné merania pomocou nástroja k6 ukázali, že monolitická architektúra dosahuje o 14\,\% vyšší peak throughput (1\,110 oproti 972 req/s), 2,2-násobne nižšiu mediánovú latenciu HTTP a výrazne kratší end-to-end čas spracovania (5\,ms oproti 5\,210\,ms medián). Dominantným faktorom tohto rozdielu je asynchrónny outbox scheduler, nie surový výpočtový výkon. Kvalitatívna analýza potvrdila, že mikroservisy poskytujú lepšiu izoláciu zlyhaní, nezávislú nasaditeľnosť a potenciál pre selektívne škálovanie za cenu výrazne vyššej prevádzkovej zložitosti. Výsledky podporujú kontextovo závislé architektonické rozhodovanie namiesto univerzálnej preferencie jedného prístupu.
